目标是将当前原生 Android 本地音乐播放器迁移为 Flutter/Dart 跨平台应用，在保留 Android 现有能力的同时，为 iOS、macOS 和后续更多平台预留清晰边界。

结论

建议采用“新建 Flutter 工程 + 逐功能重建 + 原 Android 逻辑作为行为规格”的迁移方式，而不是把 Java 代码逐行翻译成 Dart。现有工程核心集中在一个 1672 行的 MainActivity.java 中，播放、扫描、收藏、队列、媒体会话、UI 动效耦合较重。直接翻译会把这些耦合带到 Flutter，后续迁 iOS 会继续受阻。

第一优先级：跨平台播放内核 用 just_audio 管播放与队列，用 audio_service 对接后台播放、通知、锁屏和耳机控制，用 audio_session 统一音频焦点与中断处理。
第二优先级：媒体库与目录访问抽象 Android 保留 MediaStore/SAF 能力；iOS 使用系统媒体库权限和文件选择。把平台差异收敛在 repository/plugin 层，UI 和播放状态不直接碰平台 API。
第三优先级：复刻视觉，不复刻 XML Android XML drawable 和自定义 View 应迁移为 Flutter 主题 token、CustomPainter、AnimationController 和组件，而不是资源一比一搬运。

现状盘点

模块 当前实现 迁移判断
播放 MediaPlayer、AudioManager、MediaSession 均在 MainActivity 内处理。 重建为独立 AudioHandler，避免 UI 生命周期释放播放器。
后台播放 PlaybackService 只负责前台通知，播放状态仍由 Activity 控制。 Flutter 版应让后台服务成为播放入口，UI 只订阅状态。
媒体扫描 系统曲库走 MediaStore，目录选择走 ACTION_OPEN_DOCUMENT_TREE 和 DocumentsContract 递归。 做 MusicLibraryRepository 抽象，Android/iOS 分平台实现。
收藏 SharedPreferences 保存 favorite key，包含 v2 key 和 legacy uri key 兼容。 可迁为 shared_preferences 或轻量数据库，保留 key 兼容策略。
播放队列 全部歌曲、收藏队列、随机、列表循环、单曲循环在 Activity 内维护。 抽成 PlaybackQueueController，单元测试覆盖切歌规则。
视觉 三套样式：霓虹、波纹、雷达；自定义 ThemeVisualizerView 绘制波纹和雷达。 Flutter 用 CustomPainter 和动画控制器复刻，视觉资源只作参考。
工程结构 原生 Java 单模块；目标 dart 目录当前为空。 适合从干净 Flutter 项目开始建立正确分层。

目标架构

Flutter 工程建议按 feature-first + 明确平台边界组织。播放器是产品核心，应作为应用级服务初始化；媒体库、偏好设置和可视化分别独立，避免一个页面承载全部逻辑。

分阶段开发计划

阶段 0：工程基线与规格冻结（0.5-1 天）

在 dart 目录创建 Flutter 工程，固定包名、应用名、最低 SDK、lint、目录结构和 CI 命令。把原 Android README 中的功能清单转为验收清单。

验收：flutter analyze、空 App 可运行；文档记录 Android 现有行为和迁移优先级。

阶段 1：领域模型与播放队列（1-2 天）

定义 Song、PlaybackMode、PlaybackQueueState，迁移收藏 key 规则、格式化工具和切歌算法。先用 mock 曲库完成单元测试。

验收：随机、循环、单曲循环、收藏队列、取消收藏后的队列刷新均有单元测试。

阶段 2：播放服务 MVP（2-4 天）

接入 just_audio、audio_service 和 audio_session。实现播放、暂停、seek、上一首、下一首、后台通知、耳机/锁屏控制、播放状态流。

验收：Android 真机锁屏可控制；来电或其他 App 抢占音频时可暂停/恢复；Activity 重建不丢播放状态。

阶段 3：曲库扫描 MVP（2-3 天）

先实现系统媒体库读取。Android 可通过 on_audio_query 或平台通道查询 MediaStore；iOS 评估 on_audio_query 的 MPMediaQuery 链路。保留扫描数量上限和错误提示。

验收：授权流程完整；能读取标题、艺术家、时长、大小或可用替代字段；无权限时展示可恢复状态。

阶段 4：播放器 UI 与霓虹主题（3-5 天）

实现 Flutter 主界面、进度条、底部控制区、设置面板、歌单面板、收藏按钮和霓虹唱片动画。使用 Material/Cupertino 适配安全区，不直接搬 XML 布局。

验收：常见 Android/iPhone 尺寸无溢出；播放时动画流畅；歌单和设置面板交互完整。

阶段 5：收藏、主题与本地偏好（1-2 天）

接入 shared_preferences，保存收藏 key、主题、上次目录、上次播放模式。保留原 Android 的 v2 favorite key，必要时兼容 legacy URI key。

验收：重启后收藏和主题不丢；老 key 可迁移；收藏列表播放仍优先在收藏队列中切换。

阶段 6：目录选择与递归扫描（2-5 天）

评估 file_picker.getDirectoryPath() 是否满足 Android/iOS 真实目录访问。如果需要 Android SAF tree URI 持久授权和递归 DocumentFile 读取，则补一个小型平台通道。

验收：选择目录后能递归识别 mp3、m4a、aac、wav、flac、ogg、opus 等格式；目录权限重启后仍可用。

阶段 7：波纹/雷达视觉迁移（2-4 天）

把 ThemeVisualizerView 的波纹点阵和雷达扫描迁成两个 CustomPainter。动画状态由播放状态驱动，非播放时降频，减少电量消耗。

验收：三套主题可切换；低端 Android 设备 60fps 接近稳定；iOS 渲染无明显卡顿。

阶段 8：iOS 适配与发布准备（3-6 天）

补齐 Info.plist 的媒体库用途说明、后台音频能力、控制中心行为和权限文案。检查 App Store 对本地媒体访问和后台播放的审核风险。

验收：iPhone 真机可授权、播放、后台控制、锁屏控制；无权限路径有清晰提示。

风险与应对

高风险 iOS 本地音乐访问不等同于 Android 文件扫描

Android 可以通过 MediaStore/SAF 读取本地文件 URI；iOS 更多依赖媒体库权限、文件选择器和安全作用域。不要把“选择任意目录递归扫描”作为 iOS MVP 的强承诺。

应对：先把媒体库读取作为跨平台 MVP；目录递归作为 Android 强能力、iOS 作为文件导入/目录选择实验项。

高风险 后台播放生命周期

原工程播放对象在 Activity 内，切后台和进程回收存在天然边界。Flutter 版必须从一开始使用 audio_service 的 handler 模型。

应对：阶段 2 就做后台播放，不等 UI 完成后再补。

中风险 权限与商店审核

Android 13+ 需要使用音频媒体权限；全盘文件权限属于高风险。iOS 需要明确媒体库用途说明。

应对：避免申请 MANAGE_EXTERNAL_STORAGE；权限弹窗文案聚焦“读取本机音频用于本地播放”。

中风险 可视化耗电与掉帧

波纹和雷达当前依赖 Canvas 高频绘制。Flutter 迁移后如果每帧重绘面积过大，会影响低端机。

应对：使用 RepaintBoundary、动画降频、播放暂停状态分离，并用 profile 模式实测。

中风险 依赖插件版本与平台差异

音频插件通常涉及 Android ExoPlayer、iOS AVFoundation、后台服务和权限配置，升级时容易出现平台侧冲突。

应对：锁定版本，建立 Android/iOS 真机 smoke test；每次升级插件单独验证。

可控 收藏数据迁移

当前收藏 key 规则简单，迁移难度低。

应对：保留 v2|title|duration|size 规则，必要时读取 legacy URI key 做一次性迁移。
